# Frontend Staged Commit Message

## Overview

Generate exactly one concise English commit message from frontend changes that are already staged with `git add`.

This skill is message-only. It never stages files, creates commits, edits files, runs formatters, runs fixes, or performs code review. It reads the staged diff, infers the dominant frontend intent, and returns a single line that the user can paste into `git commit -m`.

Core output shape:

```text
<type>: <concise English frontend change description>
```

Do not use Conventional Commit scope parentheses. Put the page, component, API, route, or shared module context inside the description instead.

Good:

```text
feat: add bulk selection actions to order list
fix: preserve login redirect target after sign-in
style: align avatar layout on user profile page
```

Bad:

```text
feat(order-list): add bulk selection actions
fix(login): preserve redirect target after sign-in
style(user-profile): align avatar layout
```

## When to Use

Use when the user says any of:

- "generate a commit message for staged frontend changes"
- "幫我根據 git add 的內容產生 commit"
- "產生一個單行英文 commit"
- "frontend staged commit message"
- "read git diff --cached and write a commit message"

Do not use for:

- Actually running `git commit`.
- Staging files with `git add`.
- Reviewing staged changes for defects; use `frontend-staged-review-workflow` for review.
- Debugging or fixing code.
- Multi-line release notes, PR descriptions, or changelog entries.
- Backend-only changes with no frontend impact.

## Core Contract

1. Read only already staged changes as the commit-message source.
2. Use `git status --short --branch` only to understand repository state and mark unstaged files out of scope.
3. Do not use unstaged `git diff` content to decide the message.
4. Do not run `git add`.
5. Do not run `git commit`, `git commit --amend`, formatters, linters, tests, builds, or auto-fixes.
6. Output exactly one English line by default.
7. Do not wrap the output in Markdown or a fenced code block.
8. Do not add a body, footer, bullet list, explanation, issue reference, or co-author trailer unless the user explicitly asks for a different format.
9. Never include AI/tool attribution such as `Generated by`, `Co-Authored-By`, `Claude`, `ChatGPT`, `Hermes`, model names, or agent names.
10. If the staged changes are empty, stop and say that nothing is staged.
11. If staged changes contain unrelated intents, recommend splitting commits instead of forcing a misleading single line. If the user insists on one line, choose the dominant intent and avoid overclaiming.

## Required Git Inspection

Run and capture internally:

```bash
git rev-parse --show-toplevel
git status --short --branch
git diff --cached --name-only
git diff --cached --stat
git diff --cached
git log --format=%s -n 20
```

Rules:

- If `git diff --cached --name-only` is empty, stop.
- Treat unstaged files from `git status --short --branch` as out of scope.
- If the staged diff is very large, inspect the staged file list and stat first, then read the relevant staged diff chunks needed to infer intent.
- Recent commit subjects are context only. Follow this skill's no-scope output rule even if history uses `type(scope): subject`.
- If local convention files exist, read them only for type vocabulary and wording style; do not adopt scoped headers unless the user overrides this skill.

Likely convention files:

```text
COMMIT_CONVENTION.md
CONTRIBUTING.md
.github/COMMIT_CONVENTION.md
.github/CONTRIBUTING.md
.commitlintrc*
commitlint.config.*
```

## Output Format

The final answer should normally be exactly one line:

```text
<type>: <description>
```

Constraints:

- English only.
- Imperative mood: `add`, `fix`, `preserve`, `align`, `extract`, `simplify`.
- Lowercase description unless it starts with a proper noun, package name, or code identifier.
- No trailing period.
- Target 72 characters or less when possible; prefer clarity over an artificial abbreviation.
- No parenthesized scope after the type.
- Include frontend context in the description, not in the header.

Examples:

```text
feat: add status filter to user management table
fix: handle empty payloads in order API client
style: refine filter panel spacing on report page
refactor: extract empty value normalization into shared formatter
perf: reduce user table re-renders during filter changes
```

## Type Selection

Choose the type from the dominant staged intent:

| Type | Use when staged changes primarily... | Example |
|---|---|---|
| `feat` | add user-visible behavior, UI capability, route, interaction, API request capability, or form/table feature | `feat: add bulk selection actions to order list` |
| `fix` | correct broken behavior, edge cases, API response handling, state bugs, route bugs, form bugs, or rendering issues | `fix: preserve login redirect target after sign-in` |
| `style` | change visual appearance, layout, spacing, CSS, Tailwind classes, design tokens, or Ant Design styling without behavior change | `style: align toolbar spacing on user table` |
| `refactor` | restructure code without changing behavior, extract helpers, simplify state mapping, move shared logic | `refactor: extract date formatting into shared helper` |
| `perf` | improve frontend performance, reduce re-renders, optimize grid rendering, memoize hot paths | `perf: reduce order table cell renderer updates` |
| `docs` | change documentation only | `docs: document staged commit message workflow` |
| `test` | change tests only | `test: update login redirect coverage` |
| `build` | change build config, bundling, Vite, package scripts, path aliases, or build dependencies | `build: add Vite path alias resolution` |
| `ci` | change CI/CD pipeline behavior only | `ci: run frontend typecheck on pull requests` |
| `chore` | perform maintenance, dependency updates, repository housekeeping, or tooling changes with no product behavior | `chore: update frontend lint configuration` |

When multiple types seem possible, prefer user-visible intent over implementation detail. For example, a TypeScript refactor that enables a new visible filter should usually be `feat`, not `refactor`.

Breaking changes may use `!` without a scope:

```text
feat!: remove legacy account settings route
fix!: change user API error handling behavior
```

Use `!` only when the staged diff clearly changes a public contract, route, exported API, required prop, config option, or migration behavior. Do not guess.

## Frontend Context Classification

Infer the context from paths, exports, route definitions, component names, API client names, tests, and changed symbols. Write that context inside the description.

| Evidence | Context phrase examples | Example output |
|---|---|---|
| route/page files, route config, `pages/`, `app/`, screen components | `on login page`, `to order list`, `on user profile page` | `fix: preserve login redirect target after sign-in` |
| React component or UI module | `in user table`, `to date range filter`, `in filter panel` | `feat: add clear button to date range filter` |
| visual-only CSS/Tailwind/token changes | `on dashboard cards`, `in report page`, `on settings form` | `style: refine responsive spacing on dashboard cards` |
| API client, service, request params, response mapping | `in user API requests`, `in order API client`, `in account service` | `fix: handle empty order detail responses in API client` |
| shared utility or helper | `in shared formatter`, `in date helper`, `in validators` | `fix: handle null values in shared date formatter` |
| React hook or state flow | `in user search hook`, `for order filters` | `fix: avoid stale filter state in user search hook` |
| Ant Design component usage | `on settings form`, `in edit modal`, `on user table` | `style: align Ant Design form spacing on settings page` |
| AG Grid config, row identity, cell renderers | `in user table`, `in order grid` | `fix: stabilize AG Grid row identity in user table` |
| React Hook Form, Zod, validation | `in profile form`, `after server validation` | `fix: preserve form errors after server validation` |
| i18n files or translation keys | `for user table`, `on order list` | `fix: add missing user table translation labels` |
| React Router navigation, params, loaders/actions | `when switching report tabs`, `to account detail page` | `fix: preserve search params when switching report tabs` |

Context priority:

1. User-facing page or route name.
2. Stable component or domain name.
3. API client, service, or request boundary.
4. Shared module such as formatter, validator, hook, or utility.
5. Package or app name in a monorepo.
6. `frontend` only when the change is truly cross-cutting.

Avoid raw file names unless they are the clearest stable module name.

## Single-Intent Gate

Before producing the final line, decide whether the staged diff represents one coherent intent.

Single intent examples:

- UI and translation changes for the same new button.
- API client and table UI changes for the same new filter.
- Component and shared helper changes for the same bug fix.

Unrelated intent examples:

- Login bug fix plus dashboard layout cleanup.
- Dependency upgrade plus unrelated form behavior change.
- Documentation update plus production code feature.

If unrelated staged intents exist, say briefly that the staged changes should be split and provide candidate lines, for example:

```text
The staged changes look like separate commits:
- fix: preserve login redirect target after sign-in
- style: refine dashboard card spacing
```

Only output one line anyway when the user explicitly asks for one combined message. In that case, choose the dominant intent and do not claim details that apply only to unrelated minor changes.

## Rejection Checklist

Reject and rewrite any candidate message that:

- Uses `type(scope): description`.
- Omits the frontend page/component/API/module context when it is available.
- Uses vague wording such as `update`, `fix bug`, `change UI`, `adjust frontend`, `modify files`, `misc changes`, `wip`, or `clean code`.
- Describes files instead of intent, such as `update UserTable.tsx`.
- Claims behavior not supported by the staged diff.
- Mentions unstaged changes.
- Includes AI/tool attribution.
- Ends with a period.
- Uses past tense such as `added`, `fixed`, or `updated`.

## Examples

| Staged intent | Good output | Bad output |
|---|---|---|
| Add order list bulk selection UI | `feat: add bulk selection actions to order list` | `feat(order-list): add bulk selection actions` |
| Fix redirect after login | `fix: preserve login redirect target after sign-in` | `fix: fix login bug` |
| CSS-only user profile avatar adjustment | `style: align avatar layout on user profile page` | `style(user-profile): update avatar` |
| Add API request filter and connect table control | `feat: include status filter in user list requests` | `feat(api): update request` |
| Extract reusable empty-value formatting | `refactor: extract empty value normalization into shared formatter` | `refactor(utils): clean code` |
| AG Grid row id bug | `fix: stabilize AG Grid row identity in user table` | `fix(ag-grid): fix rows` |
| Missing translations | `fix: add missing user table translation labels` | `fix(i18n): update translations` |
| Vite alias config | `build: add Vite path alias resolution` | `chore(vite): modify config` |

## Common Pitfalls

1. **Using scoped Conventional Commits by habit.** This skill intentionally forbids `feat(scope): ...`; put the context in the description.
2. **Summarizing file paths instead of user-visible intent.** `fix: update UserTable.tsx` is not useful history.
3. **Reading unstaged diffs.** The user staged a deliberate subset; do not mix in worktree changes.
4. **Overclaiming on mixed changes.** If the staged diff has unrelated intents, recommend splitting commits.
5. **Adding commit-message body text.** The default output is one line only.
6. **Adding AI attribution.** Commit messages describe the change, not the tool that drafted the message.

Before finalizing, confirm:

- [ ] `git diff --cached --name-only` was non-empty.

- [ ] The description includes page/component/API/module context when available.
- [ ] The type matches the dominant staged intent.
- [ ] The subject is imperative, concise, and has no trailing period.
